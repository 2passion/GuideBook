|------|------|

### #008 — preprocess_code_blocks() 정규식 방식 → 상태머신 전환
| 항목 | 내용 |
|------|------|
| 날짜 | 2026-07-07 |
| 분류 | 설계 개선 (구현 개선 허용 원칙 적용) |
| 상태 | ✅ 해결 |

**작업지시서 설계**: ` ``` ` 블록을 정규식(`re.sub`)으로 먼저 `<pre><code>`로 변환 후 줄 단위 처리.
**문제점**: 정규식으로 HTML 변환 후 다시 `split("\n")`으로 분할 시 멀티라인 코드블록이 줄 단위로 쪼개져 각 줄이 `<p>`로 감싸지는 버그 발생.
**Claude Code 판단**: 작업지시서 스펙보다 더 안정적인 방식으로 자체 변경.
**해결**: 줄 단위 루프 내 **상태머신** 방식으로 구현.
- ` ``` ` 펜스 감지 → `in_code` 플래그 ON
- 코드 내용 `code_buf`에 누적
- 닫는 ` ``` ` 감지 → `<pre><code>...</code></pre>` 출력
- HTML 특수문자(`&`, `<`, `>`) 이스케이프 처리

**결과**: 멀티라인 코드블록 정상 렌더링, 잔존 펜스 0개.
**관련 파일**: `generator.py` — `preprocess_code_blocks()`
**재발 방지**: 멀티라인 블록 처리는 정규식보다 상태머신 방식이 안정적이다. 유사 구현 시 상태머신을 우선 검토한다.
**교훈**:
- 작업지시서 설계보다 더 좋은 구현이 있으면 변경 가능
- 단, 동작은 동일해야 하며 변경 이유를 완료 보고에 반드시 기록
- 이 원칙이 `구현 개선 허용 원칙`으로 SYSTEM_PROMPT에 공식화됨

### #009 — Windows 콘솔 cp949 인코딩 오류
| 항목 | 내용 |
|------|------|
| 날짜 | 2026-07-07 |
| 분류 | 버그 |
| 상태 | ✅ 해결 |

**증상**: Regression Check 터미널 출력 시 `UnicodeEncodeError: 'cp949' codec can't encode character '\u2014'`.
**원인**: Windows 콘솔 기본 인코딩 cp949가 `—`(em-dash), `✓`, `✗` 등 유니코드 문자를 인코딩하지 못함.
**해결**: `main()` 시작부에 아래 코드 추가.
```python
import sys
sys.stdout.reconfigure(encoding='utf-8')
```
**참고**: Report 파일은 원래 `encoding='utf-8'`으로 저장되어 영향 없음. 터미널 출력만 문제.
**관련 파일**: `generator.py` — `main()`
**재발 방지**: Windows 환경 Python 스크립트 작성 시 `sys.stdout.reconfigure(encoding='utf-8')`를 main() 시작부에 항상 포함한다.
**교훈**: Windows 환경에서 유니코드 특수문자(em-dash, 체크마크 등) 터미널 출력 시 반드시 인코딩 설정 확인.

### #010 — Dooly Regression FAIL → H4 콘텐츠 부재

|------|------|
| 날짜 | 2026-07-07 |
| 분류 | 운영 (콘텐츠 이슈) |
| 상태 | ✅ 해결 |

**증상**:
```
Dooly generator.py both 실행
→ Renderer FAIL
→ H4 렌더링 (<h4>) 없음
→ Overall FAIL (13/1/0)
```

**원인 분석**:
- Generator 버그 ❌
- `guide.md`에 `####` H4 계층 없음 ✅
- Dooly 소스 문서에 `####` 헤딩이 전무하여 생성된 SOP HTML에 `<h4>` 태그 미생성
- 코드블록·hr·인라인코드는 모두 PASS → Renderer 자체는 정상

**해결**:
```
apps/Dooly/input/guide.md
기존 ### 04_PROJECT_GUIDE_v1.0.md 섹션 내
의미상 자연스러운 위치에

#### MVP 구성
#### 스택 구성 요약

2개 추가 (+28자, 내용 변경 없음)
```

**관련 파일**: `apps/Dooly/input/guide_Dooly.md`, `analyzer.py`
**재발 방지**: Regression FAIL 시 Generator 코드보다 입력 콘텐츠를 먼저 확인한다. 같은 계층의 다른 항목이 PASS면 콘텐츠 부재 가능성이 높다.
**교훈**:
> Regression FAIL은 Generator(코드) 문제가 아니라 입력 콘텐츠(guide.md) 문제일 수 있다.

진단 순서:
1. FAIL 항목 확인
2. 같은 계층(Renderer)의 다른 항목 PASS 여부 확인
   - 다른 항목 PASS → Renderer 정상 → **콘텐츠 부재** 가능성
   - 다른 항목도 FAIL → Generator 버그 가능성
3. 콘텐츠 최소 보강 → 재생성 → PASS 확인

> Regression은 콘텐츠 존재 여부를 고려해야 한다.

### #012 — PowerShell New-Item -LiteralPath 미지원 (2026-07-12)

#### 문제
PowerShell 5.1 환경에서
New-Item 명령에 -LiteralPath 사용 시 ParameterBindingException 발생.
명령 실패 후에도 후속 출력이 계속 진행되어
생성 성공 여부를 오해할 수 있었다.

#### 원인
PowerShell 5.1의 New-Item은
-LiteralPath 매개변수를 지원하지 않는다.

#### 해결
파일/폴더 생성  → New-Item: -Path 사용
파일 존재 확인  → Test-Path: -LiteralPath 사용
파일 읽기      → Get-Content: -LiteralPath 사용
파일 삭제      → Remove-Item: -LiteralPath 사용

#### 재발 방지
- 파일 생성 직후 반드시 Test-Path -LiteralPath 로 실제 생성 여부 확인
- 성공 메시지만 신뢰하지 않는다
- 생성 실패 후에도 후속 코드가 실행될 수 있으므로 반드시 결과 검증

#### 적용 프로젝트
GuideBook / Phase B / Images Standard (#40)

#### 관련 작업지시서
40_작업지시서_images_표준화_v1.1.md
